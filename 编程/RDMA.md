# RDMA

## 1.RDMA体系结构概述

### 1.1 InfiniBand

- InfiniBand（IB）是一种服务器和存储器的互联技术，它具有高速， 低延迟， 低CPU负载，高效率， 可扩展的特性。
- InfiniBand的关键特性之一是它天然支持远程直接内存访问（RDMA）（服务器到服务器， 服务器到存储设备之间不需要CPU）
- InfiniBand使用I/O通道（每个主机最多可有16M个）进行数据传输，每个I/O提供虚拟的NIC或HCA语义

### 1.2 虚拟协议互联（VPI）

- Mellanox的VPI架构为同时支持IB和Ethernet的网络提供了高性能， 低延迟和高可靠性。
- 一个VPI适配器或交换机可以通过使用IB语义或者Ethernet语义在每个端口设置传输方式。

#### 1.2.1 VPI（虚拟互联协议）是什么

- 是Mellanox开发的一种创新网络架构，它允许同一硬件平台同时支持Infiniband和Ethernet两种网络协议

**核心思想**：

- **硬件统一：**使用相同的芯片，网卡， 交换机硬件。
- **协议可选：**通过软件配置选择运行IB或者Ethernet协议
- **性能无损：**保持IB的高性能和低延迟特性

**功能支持**

VPI架构支持两大远程直接内存访问技术

**InfiniBand RDMA**

- 原生RDMA支持
- 零拷贝数据传输
- 内核旁路支持

**RoCE（RDMA over Converged Ethernet）**

- 在Ethernet上实现RDMA
- 保持RDMA性能优势
- 兼容现有Ethernet基础设施

### 1.3 基于RoCE的RDMA

1. RoCE是基于以太网的RDMA的技术标准，不需要复杂低效的TCP传输
2. RoCE是现在最有效的以太网低延迟方案， 它消耗很少的CPU负载，它在数据中心桥接以太网中利用优先流控制（PFC）来达到网络的无损连通。

### 1.4 不同RDMA技术的比较

- RDMA技术使用操作系统仅仅建设一个通道，然后就可以在不需要操作系统的干预下应用，程序之间能够进行直接的消息传递。
- RDMA利用栈旁路和零拷贝技术提供了低延迟的特性，同时减少了CPU占用，减少了内存带宽瓶颈，提供了很高的带宽利用率。
- IB和RoCE都支持多播模式

1. IB在链路层提供的特性有：

   - 基于信任的流控制机制用来进行拥赛控制。

   - 支持使用虚拟局域网（Vls），虚拟局域网能够使高层协议简单化，并提供高质量服务（QoS）。VL中严格保证数据在一条路径中能够按序到达。

2. IB在网络层拥有的特性：

   - 可以很简单地在应用程序的虚拟内存之间传递消息。
   - **重要：**每个应用程序都能直接访问集群中的设备的虚拟内存。意味着：应用程序 传输消息时不需要像操作系统发出请求。
   - 在IB中一个完整的消息被直接发送到一个应用程序。一旦一个应用程序请求了RDMA的读或写传输，IB的硬件将需要传输的数据按照需要分割成一些数据包（**取决于网络路径最大传输单元**），这些数据包通过IB网络发送到接收程序的虚拟内存中，当整个消息到达时接收程序会收到提示，这样发送程序和接收程序在直到整个消息被发送到达接收程序的缓冲区之前都不会被打扰中断。

### 1.5 主要构件

#### 1.5.1 Host Channel Adapter(主机通道适配器)

**HCA**为一个IB终端节点（例如一个服务器）连接到IB网络提供了连接点。这类似于以太网卡（NIC）

- HCA在操作系统的控制下提供了地址翻译机制，允许应用程序直接使用HCA。
- 利用这种地址翻译机制，HCA能够在用户态程序下堆内存进行使用。
- 应用程序直接使用虚拟地址进行操作，HCA能够将这些虚拟地址翻译为物理地址

#### 1.5.2 Range Extenders（范围扩展器）

IB将传输通路加入到WAN中，来实现的网络范围扩展。同时，增加了足够的buffer credit（BBC）来确保在WAN上能够满带宽运行。

#### 1.5.3 Subnet Manager(子网管理)

- IB子网管理器给每个连接到子网中的端口分配一个局部ID（LID），并且基于这些LID，构建了一个路由表。
- IB子网管理器是一个软件定义网络（SDN）的概念，SDN能够消除网络连接的复杂性。

### 1.5.4 Switches

IB交换机的设计是为了迎合IB性能的需求

- IB交换机就在IB链路层实施了流控制
  - 为了防止数据包丢失
  - 为了支持拥塞控制和自适应路由

很多IB交换机都包含一个子网管理器。

## RDMA基本元素

### WQ（工作队列）

Work Queue。WQ是一个存储工作请求的队列。

#### WQE（任务书）

​	WQE可以认为是一种“任务说明”，这个工作请求是软件下发给硬件的， 这份说明包含了软件所希望硬件去做的任务以及有关这个任务的详细信息。

- **WQ**就是存放 **WQE**的文件夹。
- **WQ**这个队列总是有软件向其中增加 **WQE**，硬件从中取出 **WQE**(这就是软件给硬件“下发任务“的过程 )

#### SQ和RQ

任何通信过程都要有收发两端，QP就是一个发送工作队列和一个接受工作队列的组合，这两个队列分别称为 **SQ**（Send Queue）和 **RQ**（Receive Queue）。

## RDMA操作类型

### 双端操作

#### SEND&RECV

Send和Recv是两种不同的操作类型，但是如果一端进行Send操作，对端必须进行Recv操作。

在Send&Recv操作中完成一次通信过程需要两端CPU的参与，并且收端需要提前显式的下发WQE。如图所示为一次SEND-RECV操作

### 单端操作

#### WRITE

Write全称为RDMA WRITE操作，是本端主动写入远端内存的行为，除了准备阶段，远端CPU不需要参与，也不感知何时有数据写入，数据在何时接收完毕。所以这是一种单端操作。

如图所示：

图

对比Send-Recv操作差异，本段在准备阶段通过数据交互，获取了对端某一片可用的内存的 **地址**和 **钥匙**，相当于获得了这篇远端内存的读写权限。拿到权限之后，本段就可以像访问自己内存一样直接对这一远端内存区域进行读写。

 ***目的地址和钥匙是如何获取的***？

通常可以通过 **SEND-RECV**操作来完成的，因为拿到钥匙这个过程总归是要由远端内存的控制着——CPU允许的，

- 本端是通过虚拟地址来读写远端内存的，上层应用可以非常方便的对其进行操作。实际的 **虚拟地址——物理地址**的转换是由RDMA网卡完成的。

- 是本端主动读取远端内存的行为。远端CPU不需要参与，也不感知数据在内存中被读取的过程。
- “读”这个动作所请求的数据，实在对端回复的报文中携带的

- RDMA WRITE和READ操作就是在利用网卡完成下面左图的内存拷贝操作而已，只不过复制的过程是由RDMA网卡通过网络链路完成的；

# LID和GID

InfiniBand定义的LID是一个16位的标识符。

- 由子网管理员分配，子网内唯一，不可用于子网间路由。
- LID作为一种网络地址，分为预留，单播和多播地址段。
- 数据包中的本地路由包头中包含了LID。
- 一个单播型的目的LID适用于某一个目的终端端口。一个多播型的目的LID适用于一个子网中某个多播组里的一系列目的终端端口。
- 如果最终目的端口不在这个子网内，数据包中的目的LID指向的是，负责转发这个数据包到下一跳的路由器的某个端口。
- 一个终端端口在连接到子网后，收到的数据包可能经过了子网内的多条物理路径。
- LIDded分配规则：LID　０x0000无效；LID 0xFFFF分配给接收数据包的终端端口QP0；0x0001和0xBFFF之间为单播LID； 0xC000和0xFFFE之间为多播LID。

对于RoCE 类型的网络，LID无效，所有端口的LID都为0x0000.

**GID**

是一种128位的单播或多播标识符，用于标识端口或多播组

**GUID**是全局唯一的EUI-64标识符，共64位。其中的24位标识厂商ID，另外40位是扩展标识符，由生产设备的厂商来分配。

# RDMA编程实例

首先注册内存块，然后交换内存描述符，然后发布读/写操作。

注册时通过调用来完成的，改调用将内存块固定到位（从而防止它被交换），并返回一个包含允许访问注册内存的密钥。

该密钥及区块地址必须通过某种带外机制与对等方交换，然后，对等方可以再调用中使用密钥和地址来发布RDMA读写请求

主机一：

```C
const size_t SIZE = 1024

char *buffer = malloc(SIZE);
struct ibv_mr *mr;
uint32_t my_key;
uint64_t my_addr;

mr = ibv_reg_mr(pd, buffer, SIZE, IBV_ACCESS_REMOTE_WRITE);

my_key = mr->rkey;
my_addr = (uint64_t)mr_addr;
//交换my_key 和 my_addr 和主机二
```

主机二

```
const size_t SIZE = 1024;

char *buffer = malloc(SIZE);
struct ibv_mr *mr;
struct ibv_sge sge;
struct ibv_send_wr wr, *bad_wr;
uint32_t peer_key;
uint64_t peer_addr;

mr = ibv_reg_mr(pd, buffer, SIZE, IBV_ACCESS_LOCAL_WRITE);
//获取peer_key and peer_addr from 主机一
strcpy(buffer, "Hello!");

memset(&wr, 0, szieof(wr));

sge.addr = (uint64_t)buffer;
sge.length = SIZE;
sge.lkey = mr->lkey;

wr.sg_list = &sge;
wr.num_sge = 1;
wr.opcode = IBV_WR_RDMA_WRITE;

wr.wr.rdma.remote_addr = peer_addr;
wr.wr.rdma.rkey = peer_key;

ibv_post_send(qp, &wr, &bad_wr);
```

对于对等方1的最后一个参数，指定我们希望对等方2对与1的内存块具有写入访问权限。ibv_reg_mr（）ibv_ACCESS_REMOTE_WRITE buffer

## 步骤

1. 初始化上下文和注册内存区域。
2. 建立连接
3.   
4. 发布读/写操作
5. 断开连接

连接的每一段都有两个线程：处理连接时间的主线程和轮询完成队列的线程，为了避免死锁和竞争条件，我们安排了操作，以便一次只有一个线程发布工作请求。如下：

1. 在消息中发送其RDMA内存区域键。MSG_MR
2. 等待服务器的消息包含其RDMA密钥。MSG_MR
3. 发布RDMA操作
4. 通过发送消息向服务器发出信号，表明它已经准备好断开连接。MSG_DONE
5. 等待来自服务器的消息。MSG_DONE
6. 断开

第一步发生在RDMA连接事件处理程序线程的上下文中，但步骤２到第６步发生在谓词CQ轮询线程的上下文中。服务器的操作顺序类似：

1. 等待客户端的消息机器RDMA密钥。MSG_MR
2. 在消息中发送其RDMA密钥。MSG_MR
3. 发布RDMA操作
4. 通过发送消息向客户端发出信号，表明他已准备好断开连接。MSG_DONE
5. 等待来自客户端的消息。MSG_DONE
6. 断开

在这里，所有六个步骤都发生在动词CQ轮询线程的上下文中。等待是必要的，否则我们可能会在对等方的RDMA操作完成之前关闭连接。请注意，我们不必等待RDMA操作完成即可发送——InfiniBand规范要求请求将按照发布顺去发起。这意味着在RDMA操作完成之前，对等方不会收到。MSG_DONEMSG_DOWNMSG_DONE
